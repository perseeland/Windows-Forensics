USN Journal

Overview ▼

$Extend\$UsnJrnl:$J — Root of the NTFS volume
  • The USN Journal (Update Sequence Number Journal) is an NTFS feature that records every change made to files and directories on the volume.
  • Every file creation, modification, rename, deletion, and attribute change generates a USN entry.
  • The journal is stored in a hidden system file at $Extend\$UsnJrnl and contains two data streams:
$J — Journal data stream
  • The main stream containing all change records — this is the file parsed for forensic analysis.
$Max — Journal metadata stream
  • Contains configuration info about the journal such as maximum size and allocation delta.
The USN Journal is circular — once it reaches its maximum size, older entries are overwritten. The further back in time you investigate, the less likely the entries are still present.

What It Records ▼

  • Each USN entry contains the following fields:
USN (Update Sequence Number)
  • A unique incrementing number assigned to each change record — can be used to determine order of events.
Timestamp
  • The exact date and time the change occurred — useful for timeline reconstruction.
File Name & Parent Directory
  • The name of the file and its parent folder at the time of the change.
Reason Flags
  • Describes what type of change occurred. Key flags include:
  • FILE_CREATE — file was created
  • FILE_DELETE — file was deleted
  • DATA_EXTEND — file content was added to
  • RENAME_OLD_NAME / RENAME_NEW_NAME — file was renamed
  • CLOSE — file handle was closed (often final entry for a session)
File Reference Number
  • Links the USN entry to its MFT entry — useful for correlating USN records with MFT data.

Deleted Files ▼

  • The USN Journal is one of the most valuable sources for proving file deletion — a FILE_DELETE reason flag records when a file was deleted along with its name and timestamp.
  • Critically, USN entries survive even after the MFT entry has been reused — meaning you can find evidence of deleted files that are no longer recoverable from the MFT.
  • You can reconstruct the full path of a deleted file by tracing the parent directory reference through the journal entries.
  • Look for a sequence of entries ending in RENAME_OLD_NAME → RENAME_NEW_NAME → FILE_DELETE — this is the typical pattern when a file is moved to the Recycle Bin and then permanently deleted.
  • Combining USN Journal entries with $MFT records and $I30 directory indexes gives the most complete picture of file deletion activity.
Even if a file's content is unrecoverable, the USN Journal can still prove it existed, was accessed, and was deleted — which may be sufficient evidence in an investigation.

MFTECmd ▼

  • MFTECmd by Eric Zimmermann can also parse the $J data stream of the USN Journal and export it as a CSV for analysis in Timeline Explorer.
  • The $J file must be collected from the image — KAPE's KapeTriage package collects it automatically.
  • Using the -m flag to point to the $MFT allows MFTECmd to resolve full file paths from the MFT for each USN entry — without it, only the file name is shown. Always use -m for a more complete output.
  • In Timeline Explorer filter by the Reason column — sort or search for FILE_DELETE entries to focus on deletions during the attack window.
MFTECmd
Parse $J with $MFT for full path resolution (recommended):
MFTECmd.exe -f "C:\Cases\C\$Extend\$J" -m "C:\Cases\C\$MFT" --csv "C:\Cases\Analysis\NTFS" --csvf UsnJrnl.csv
Parse $J without $MFT (file name only):
MFTECmd.exe -f "C:\Cases\C\$Extend\$J" --csv "C:\Cases\Analysis\NTFS" --csvf UsnJrnl.csv
