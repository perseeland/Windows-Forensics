Startup Folders

Overview ▼

  • Windows Startup Folders are special directories — any file placed inside them is automatically executed when a user logs in.
  • There are two variants: a per-user startup folder (only runs for that specific user) and an all-users startup folder (runs for every user who logs in).
  • Files that execute from startup folders include .exe, .bat, .vbs, .ps1, .cmd, and .lnk shortcuts pointing to other executables.
  • Unlike Run Keys, startup folder persistence is file-system based — the malicious file must be present in the folder for it to execute.
  • Attackers commonly drop .lnk shortcut files into the startup folder pointing to malware elsewhere on the system, making detection slightly harder.

Folder Locations ▼

Per-User Startup
C:\Users\<username>\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup
  • Only executes for the specific user when they log in.
  • Does not require admin privileges to write — a common attacker target.
  • Check this folder for every user profile found on the system.
All Users Startup
C:\ProgramData\Microsoft\Windows\Start Menu\Programs\StartUp
  • Executes for every user who logs into the system.
  • Requires admin privileges to write — indicates a higher-privilege attacker or escalation before placing persistence.

Registry Keys ▼

  • The actual paths of the startup folders are configured in registry keys — attackers can redirect these keys to a different folder as a stealthier persistence technique.
HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\User Shell Folders
HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\Shell Folders
  • Controls the per-user startup folder path — found in NTUSER.DAT.
  • Look for the Startup value — if it points to an unusual directory, the startup folder has been hijacked.
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\Shell Folders
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\User Shell Folders
  • Controls the all-users startup folder path — found in the SOFTWARE hive.
  • Look for the Common Startup value — a redirect here affects all users on the system.
If the Startup or Common Startup registry values point to a non-standard path, the attacker may have redirected the startup folder itself — investigate the new path for malicious files.

Investigation ▼

  • Browse the startup folder paths directly in the mounted image — check for any unexpected files or shortcuts present.
  • For .lnk shortcut files, examine the shortcut target path — it may point to malware in a completely different location on disk.
  • Use $MFT analysis to check the created and modified timestamps of files in the startup folders — cross-reference with the attack timeframe.
  • Even if the startup file has been deleted, the $MFT and USN Journal may still record that it existed and was removed.
  • In Timeline Explorer, open the $MFT CSV output from MFTECmd and filter the ParentPath column for \Startup — this will surface any files (including deleted ones) that existed in either the per-user or all-users startup folders, regardless of whether they are still present on disk.
  • In Registry Explorer, use the Bookmarks tab to navigate directly to the Shell Folders keys and verify the startup folder paths are pointing to expected locations.

RegRipper ▼

  • The shellfolders plugin outputs the configured startup folder paths from both the SOFTWARE and NTUSER.DAT hives — useful for detecting redirected startup folders.
  • Run against both hives to get a complete picture.
RegRipper
All-users startup folder path (HKLM):
rip.exe -r C:\Cases\Analysis\Registry\SOFTWARE -p shellfolders > C:\Cases\Analysis\Registry\RR_Output\shellfolders_hklm.txt && type C:\Cases\Analysis\Registry\RR_Output\shellfolders_hklm.txt
Per-user startup folder path (HKCU):
rip.exe -r C:\Cases\Analysis\Registry\NTUSER.DAT -p shellfolders > C:\Cases\Analysis\Registry\RR_Output\shellfolders_hkcu.txt && type C:\Cases\Analysis\Registry\RR_Output\shellfolders_hkcu.txt
